# error-log-check

ไล่อ่าน `error_log` ของทุกเว็บบนเซิร์ฟเวอร์ หา **เว็บที่พังจริง (Fatal error)** ด้วย PHP CLI รันตรงจาก GitHub

จับ "เว็บที่ล่มเงียบ ๆ" ที่การนับไฟล์มองไม่เห็น — เช่นปลั๊กอินที่ไฟล์ครบเกือบหมด แต่ขาดไฟล์ลึกข้างใน 1-2 ตัว ทำให้เกิด Fatal error ทั้งเว็บ

> อ่านอย่างเดียว 100% ไม่ลบ ไม่แก้ ไม่ย้ายไฟล์ — ปลอดภัย รันกี่ครั้งก็ได้

## ทำไมต้องมีตัวนี้

สคริปต์นับไฟล์ (`plugin-health` / `theme-health`) จับ "ไฟล์หายทั้งโฟลเดอร์ / หายเยอะ" ได้ดี แต่**จับ "ไฟล์ลึกหายบางตัว" ไม่ได้** เช่น ปลั๊กอิน better-wp-security ที่ขาดไฟล์ `core/Contracts/Import_Export_Source.php` เพียงไฟล์เดียว — ไฟล์อื่นครบ มี header ครบ แต่เว็บ Fatal error ล่มทั้งเว็บ

ตัวนี้มองจาก **อาการจริง** (error log) แทนการเดาจากไฟล์ จึงจับเว็บพังได้ทุกสาเหตุ ใช้คู่กับสคริปต์นับไฟล์จะครอบคลุมที่สุด

## คุณสมบัติ

- ไล่อ่าน error_log ทุกเว็บ หา **Fatal error** (เว็บล่มจริง)
- **เร็ว** — กรองด้วยเวลาแก้ไขไฟล์ก่อน (ข้ามไฟล์เก่า) + อ่านแค่ท้ายไฟล์ ไม่อ่านทั้งไฟล์ใหญ่
- **แปลงเวลาเป็นเวลาไทย** อัตโนมัติ (log เป็น UTC → +7)
- กรอง error เก่าออก — รายงานเฉพาะที่เกิดในช่วงวันที่กำหนด (ไม่รายงานเว็บที่แก้ไปแล้ว)
- บอก **เวลาล่าสุดที่เกิด** + **จำนวนครั้ง** + **ปลั๊กอิน/ธีมต้นเหตุ**
- จัดกลุ่มตามสาเหตุ เรียงกลุ่มที่กระทบมากสุด
- รันแบบถ่อมตัว (low CPU/IO priority) ไม่รบกวนเว็บอื่น

## วิธีใช้

```bash
URL='https://raw.githubusercontent.com/ufavisionseoteam19/error-log-check/main/error-log-check.php'

# สแกนทั้งเซิร์ฟเวอร์ (7 วันล่าสุด, Fatal error)
curl -s "$URL?v=$(date +%s)" | php

# เฉพาะบัญชีเดียว (เร็วกว่า เหมาะลองก่อน)
curl -s "$URL?v=$(date +%s)" | php -- --user=cloudwa3

# เปลี่ยนช่วงเวลาเป็น 3 วันล่าสุด
curl -s "$URL?v=$(date +%s)" | php -- --days=3

# รวม Warning ด้วย (ผลเยอะขึ้น)
curl -s "$URL?v=$(date +%s)" | php -- --warnings

# ออกผลเป็น CSV
curl -s "$URL?v=$(date +%s)" | php -- --csv > error-report-$(date +%F).csv
```

## ตัวเลือก (Flags)

| Flag | ความหมาย | ค่าเริ่มต้น |
|---|---|---|
| `--days=N` | ดู error_log ที่ถูกแก้ใน N วันล่าสุด | 7 |
| `--user=NAME` | เฉพาะบัญชีเดียว | ทุกบัญชี |
| `--base=PATH` | โฟลเดอร์ฐาน | /home |
| `--warnings` | รวม PHP Warning/Deprecated ด้วย | แค่ Fatal error |
| `--tail=N` | อ่านท้ายไฟล์กี่ KB ต่อไฟล์ | 16 |
| `--tz=N` | เขตเวลา +N สำหรับแสดงผล | 7 (ไทย) |
| `--csv` | ออกผลเป็น CSV | ตาราง |

## ตัวอย่างผลลัพธ์

```
=======================================================
 WordPress Error Log Check — หาเว็บที่พังจริง (read-only)
=======================================================
 ช่วงเวลา     : 7 วันล่าสุด
 จับ          : Fatal error เท่านั้น
 error_log    : 5626 ไฟล์
 เว็บที่พบปัญหา: 12 เว็บ

** [plugins/better-wp-security] กระทบ 8 เว็บ **
  ┌─ /home/cloudwa3/maxma88.net
  │   เวลาล่าสุด : 2026-06-06 15:32 (+7)  (เกิด 2+ ครั้ง)
  │   สาเหตุ     : Uncaught Error: Interface "...Import_Export_Source" not found
  └──────────────────────────────
```

## วิธีอ่านผล

- **เวลาล่าสุด** = ครั้งสุดท้ายที่เกิด error (เวลาไทย) — ถ้าใกล้ปัจจุบัน = ยังพังอยู่
- **เกิด N+ ครั้ง** = จำนวนครั้งที่เจอในท้าย log (มาก= มีคนเข้าเว็บเจอ error เยอะ)
- **จัดกลุ่มตามปลั๊กอิน/ธีม** = ถ้าปลั๊กอินตัวเดียวกระทบหลายเว็บ = น่าจะเป็นเหตุการณ์เดียว (เช่น auto-update พังพร้อมกัน) แก้ทีเดียวได้หลายเว็บ

## หมายเหตุเรื่องเวลา (สำคัญ)

error_log บันทึกเวลาเป็น **UTC** แต่สคริปต์แปลงเป็น **เวลาไทย (UTC+7)** ให้อัตโนมัติในการแสดงผล เช่น `08:32 UTC` จะแสดงเป็น `15:32 (+7)` — ตรงกับเวลาที่คุณเห็นบนหน้าจอ

## ข้อจำกัด

- อ่านแค่ **ท้ายไฟล์** (16 KB) เพื่อความเร็ว — ถ้า error เก่ามากอยู่ต้นไฟล์และไม่มี error ใหม่ท้ายไฟล์ อาจไม่เห็น (แต่ปกติ error ล่าสุดอยู่ท้ายเสมอ) ปรับด้วย `--tail=N` ถ้าต้องการอ่านมากขึ้น
- จับเฉพาะเว็บที่มีไฟล์ `error_log` ในโฟลเดอร์ — เว็บที่ปิด error logging หรือ log ไปที่อื่นจะไม่เห็น
- ไม่ได้บอกว่าไฟล์ไหนหาย — บอกแค่ปลั๊กอิน/ธีมต้นเหตุ ต้องดู error เต็ม ๆ ในไฟล์เพื่อรู้ไฟล์ที่ขาด

## หยุดสคริปต์ (กรณีฉุกเฉิน)

```bash
pkill -f error-log-check
```

## ความปลอดภัย

- read-only ไม่มีคำสั่งเขียน/ลบ/แก้ไฟล์เว็บ
- อ่าน error_log อย่างเดียว ไม่แตะไฟล์เว็บ

## ข้อกำหนด

- PHP CLI (ทดสอบบน PHP 8.x ใช้ได้ตั้งแต่ PHP 7.x)
- คำสั่ง `find` (มีอยู่แล้วทุกเครื่อง Linux)
- สิทธิ์อ่าน error_log (ปกติใช้ root)
